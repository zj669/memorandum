# PRD - Android 本地优先 AI 个人规划助手

## 1. 产品定位
一个仅支持 Android 的、本地优先的 AI 个人规划助手。用户手动选择内容类型，数据默认只存储在本地；AI 基于本地上下文进行任务拆解、时间安排、长期记忆归纳与主动提醒。在需要联网补充信息时，由用户配置的远程 HTTP MCP 工具提供搜索能力。

## 2. 核心目标
1. 帮助个人用户快速记录任务与背景信息。
2. 由 AI 全权完成规划、拆解、排程和提醒决策。
3. 长期积累用户画像与任务记忆，使计划越来越贴近个人习惯。
4. 通过系统本地通知主动触达用户，而不是聊天式提醒。
5. 让 AI 在必要时发起一次高价值问题补充，而不是频繁追问。

## 3. 约束与边界
### 3.1 已确认约束
- 平台仅限 Android。
- 纯前端应用，不依赖自建后端。
- 用户数据默认仅保存在本地。
- 不做多端同步。
- 不做自动分类；由用户手动选择内容类型。
- 暂不支持语音输入。
- 图片不做 OCR，直接作为图片附件提供给 AI。
- 联网能力通过用户配置的 MCP 提供。
- AI 模型由用户自行配置 API。
- 规划、拆解、排程、是否提醒，都由 AI 决定。
- APP 使用本地通知触达用户。
- 存在固定间隔的 AI 心跳机制，每次醒来决定是否通知用户。
- 用户画像依赖长期积累，单次任务最多补充提问一次。

### 3.2 非目标
- 不做账号体系。
- 不做云端备份和同步。
- 不做聊天式主界面。
- 不做规则驱动的复杂调度引擎，业务规则主要体现在 prompt 中。
- 不做 OCR、语音识别、团队协作。
- 不做自定义内容类型。

## 4. 目标用户
- 个人用户
- 需要长期规划、任务拆解、定期提醒的人
- 愿意自行配置 AI API / MCP 能力，并接受本地优先使用方式的人

## 5. 固定枚举
以下枚举在第一版中写死，不开放自定义。

### 5.1 内容类型 `EntryType`
- `TASK`：明确要执行的任务
- `GOAL`：需要长期推进的目标
- `MEMO`：普通备忘、想法、记录
- `REFERENCE`：资料、截图、参考信息
- `REMINDER`：用户明确想要提醒的事项

### 5.2 任务状态 `TaskStatus`
- `INBOX`：刚录入，尚未完成规划
- `PLANNED`：已完成 AI 规划
- `DOING`：用户正在推进
- `BLOCKED`：存在阻塞因素
- `DONE`：已完成
- `DROPPED`：已放弃

### 5.3 规划状态 `PlanningState`
- `NOT_STARTED`：尚未发起规划
- `ASKING`：等待一次补充回答
- `GENERATING`：AI 正在规划
- `READY`：规划结果可展示
- `FAILED`：本次规划失败

### 5.4 记忆类型 `MemoryType`
- `PREFERENCE`：个人偏好
- `PATTERN`：长期行为模式
- `LONG_TERM_GOAL`：长期目标
- `TASK_CONTEXT`：跨任务上下文
- `PREP_TEMPLATE`：准备步骤模板

### 5.5 通知类型 `NotificationType`
- `PLAN_READY`：新计划已生成
- `TIME_TO_START`：建议开始执行
- `DEADLINE_RISK`：截止风险提醒
- `PREP_NEEDED`：需要先准备
- `STALE_TASK`：任务长期未推进
- `HEARTBEAT_CHECK`：AI 心跳主动提醒

### 5.6 通知动作 `NotificationActionType`
- `OPEN_TASK`：打开任务详情
- `OPEN_TODAY`：打开今日计划页
- `SNOOZE`：稍后提醒
- `MARK_DONE`：快速标记完成

## 6. 用户画像策略
### 6.1 核心原则
- 用户画像不是一次性问卷产物，而是长期积累的本地记忆。
- 每次新任务或重新规划时，AI 最多可以发起 **一次补充提问**。
- 如果当前信息足够，AI 不提问，直接规划。
- 单次提问必须直接影响排程、拆解或提醒质量。

### 6.2 用户画像来源
1. 用户历史任务与完成情况
2. 用户接受/忽略/延后通知的行为
3. 用户对计划的采纳与调整
4. 用户在单次任务中的补充回答
5. 用户对任务状态的手动变更记录

### 6.3 用户画像建议字段
- 常见可用时间段
- 偏好工作时段
- 偏好任务粒度
- 对提醒频率的容忍度
- 任务拖延模式
- 常见阻塞类型
- 长期目标集合
- 任务执行习惯
- 高强度任务可承受时长
- 常见准备材料偏好

### 6.4 单次补充提问触发条件
仅当以下任一条件满足时，AI 才可提问：
- 截止时间缺失且会显著影响排程
- 预计时长缺失且任务规模明显较大
- 成功标准不明确，无法拆解为步骤
- 任务依赖或外部限制信息缺失
- 用户画像对该类任务证据不足

### 6.5 单次补充提问的产品要求
- 一次只问一个问题
- 问题必须可直接回答，避免开放式长对话
- 允许用户跳过，跳过后 AI 仍需给出保底计划

## 7. 核心功能
### 7.1 录入
- 用户创建条目时必须手选类型。
- 支持文本输入。
- 支持附加图片作为上下文。
- 支持填写截止时间、优先级、预计耗时等可选信息。
- 创建后立即本地保存，再异步进入规划流程。

### 7.2 AI 规划
AI 针对任务或目标完成：
- 任务拆解
- 准备步骤提炼
- 时间安排
- 当前优先建议
- 风险识别
- 是否需要调用 MCP 搜索
- 是否需要补充提问

### 7.3 MCP 联网能力
- 用户在设置中配置远程 HTTP MCP 服务。
- AI 只有在需要补充外部信息时才调用 MCP。
- MCP 主要用于搜索、抓取和外部知识补充。
- 若 MCP 调用失败，AI 必须提供不依赖联网的保底规划。

### 7.4 长期记忆
- AI 基于历史行为不断更新用户画像和长期记忆。
- AI 会将稳定偏好、长期目标、常见准备模式沉淀为结构化记忆。
- 记忆需要有置信度与证据来源，避免无限累积噪音。

### 7.5 心跳与通知
- 应用以固定间隔唤醒 AI。
- 每次心跳检查当前任务、用户画像、历史通知和截止风险。
- AI 决定是否通知用户以及通知内容。
- 通知通过系统本地通知触达。
- 客户端保留去重、静默时段和频率保护。

### 7.6 计划展示
- 计划结果以结构化卡片展示，不以聊天消息为主。
- 每个计划至少包含：下一步、时间块、准备项、风险、通知建议。
- 用户可以手动接受、忽略、稍后执行或标记完成。

## 8. 页面信息架构
### 8.1 Today 首页
职责：展示 AI 当前建议与今日可执行内容。

内容：
- 今日首要任务
- 推荐时间块
- 即将截止风险
- 最近一次 AI 心跳结论
- 待处理的补充问题

### 8.2 Entry 新建页
职责：让用户快速创建条目。

内容：
- 选择固定类型
- 文本输入
- 图片附件
- 可选字段：截止时间、优先级、预计时长
- 保存并触发规划

### 8.3 Tasks 列表页
职责：集中查看所有任务与目标。

内容：
- 按状态筛选
- 按截止时间排序
- 查看规划是否完成
- 进入任务详情

### 8.4 Task Detail 详情页
职责：承载单个任务的完整 AI 结果。

内容：
- 任务摘要
- 下一步动作
- 步骤拆解
- 时间安排
- 准备项
- 风险提示
- 相关图片
- 手动状态变更

### 8.5 Memory 页面
职责：查看 AI 归纳出的长期记忆。

内容：
- 长期目标
- 偏好总结
- 行为模式
- 常见准备模板

### 8.6 Notifications 页面
职责：查看 AI 曾经发出的通知和触发原因。

内容：
- 通知历史
- 是否点击/忽略
- 对应任务
- 触发原因摘要

### 8.7 Settings 页面
职责：管理模型、MCP、通知与隐私边界。

内容：
- LLM 配置
- MCP 配置
- 心跳频率
- 静默时段
- 是否允许联网
- 本地数据清理与导出（后续）

## 9. 核心用户流程
### 9.1 创建任务并规划
1. 用户进入 Entry 页
2. 手动选择类型并输入内容
3. 本地保存条目
4. AI 判断是否需要一次补充提问
5. 如需提问，则展示单个问题并等待回答
6. AI 生成计划；必要时调用 MCP
7. 结果落地到 Task Detail 与 Today

### 9.2 AI 心跳并通知
1. 后台按固定间隔唤醒
2. AI 评估当前任务、记忆与通知历史
3. 若值得提醒，则生成通知
4. 系统发送本地通知
5. 用户点击后跳转到 Today 或对应任务详情

### 9.3 长期记忆沉淀
1. 用户完成任务、调整计划或响应通知
2. 行为被本地记录
3. AI 汇总行为证据
4. 用户画像与长期记忆被更新

## 10. AI 决策边界
AI 负责：
- 是否需要补充提问
- 是否需要调用 MCP
- 如何拆解任务
- 如何排程
- 是否通知用户
- 如何更新长期记忆

应用本身负责：
- 数据持久化
- JSON 结构校验
- MCP / LLM 调用编排
- 本地通知下发
- 基础去重与频率保护
- 图片 URI、权限与本地附件管理

## 11. 我的产品决策
### 11.1 单次任务补充提问机制
- 每个任务在一次规划流程中最多触发一次提问。
- 该问题必须是高价值问题，例如：
  - 截止时间不清
  - 可用时间未知
  - 目标标准不明确
  - 任务约束缺失
- 如果回答后仍不完整，AI 也必须继续规划，而不是追问第二次。

### 11.2 通知保护机制
尽管通知内容由 AI 决定，客户端仍保留以下硬保护：
- 同一次心跳最多发出 1 条通知。
- 默认安静时段内不主动通知，除非属于高风险截止提醒。
- 短时间内重复理由的通知需要去重。
- 同一任务在短周期内重复提醒时需要冷却时间。

### 11.3 计划展示方式
计划不以聊天消息为主，而以结构化卡片展示：
- 今日建议
- 下一步动作
- 准备事项
- 时间块安排
- 风险提示

### 11.4 MCP 隐私边界
- 只有当用户开启联网能力时，AI 才允许调用 MCP。
- 默认优先发送任务摘要而不是完整历史上下文。
- 用户需要能查看最近一次 MCP 调用的目的和结果摘要。
